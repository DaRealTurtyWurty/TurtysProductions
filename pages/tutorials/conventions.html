Naming Conventions

When it comes to naming things, we want to make sure we are naming everything in our mod using the right conventions. The main reason we use these conventions is to keep some consistency within people's code, making the jump of looking at one person's code to looking at another persons code much smaller and easier to handle.

Java Naming Conventions

Type Convention Description Invalid Example Valid Example
Projects Upper Camel Case (Pascal Case) Should be a name that can easily explain what your project is for. Should always start with a capital letter; Any subsequent words should also start with a capital letter. Generally, this will be a noun. 
largevoidmod
largeVoidMod
Large_Void_Mod
LargeVoidMod
Packages Snake Case Should always be lowercase and should not contain any special symbols other than underscores (_). Should well describe what is inside of that package. The starting packages of a project should be any website you have, followed by the name of the project. For example com.google.gson. If you do not own a domain, you should use your github or similar url. For example io.github.username.projectname. 
io.github.darealturtywurty.largeVoidMod
io.github.darealturtywurty.LargeVoidMod
                            
io.github.darealturtywurty.large_void_mod
io.github.darealturtywurty.largevoidmod
Classes, Interfaces, Enums and Records Upper Camel Case (Pascal Case) Should always start with a capital letter; Any subsequent words should also start with a capital letter. Acronyms should not be used unless it would prevent an extremely long class name (any more than 24 characters is unacceptable). This should always be a noun and should well describe what this Class does. 
voideater.java
voidEater.java
Void_Eater.java
VoidEater.java
Interfaces Upper Camel Case (Pascal Case) Should always start with a capital letter; Any subsequent words should also start with a capital letter. Acronyms should not be used unless it would prevent an extremely long class name (any more than 24 characters is unacceptable).
This should always be an adjective and should describe the implication of the effects you would get from implementing.
 
supershiny.java
superShiny.java
Super_Shiny.java
SuperShiny.java
Enums and Records Upper Camel Case (Pascal Case) Should always start with a capital letter; Any subsequent words should also start with a capitalized letter. Acronyms should not be used unless it would prevent an extremely long class name (any more than 24 characters is unacceptable). This should always be an noun and should describe what the Enum or Record Contains. 
voidparts.java
voidParts.java
Void_Parts.java
VoidParts.java
Methods Camel Case Methods should always start with a lowercase letter; Any subsequent words should start with a capitalized letter. Methods should be a verb and need to very well describe what the function of such method is. Long method names should however be avoided when possible. Method names should not start with $ or numbers even though they can. 
applyvoideffect();
ApplyVoidEffect();
apply_void_effect();
apply_Void_Effect();
applyVoidEffect();
Variables Camel Case Variables should always start with a lowercase letter; Any subsequent words should start with a capitalized letter. It is very important that variables have descriptive but short names to describe the object/value that they will be holding. Any single letter variables should not be used even in the instance of loops. Variables should not start with $ or numbers even though they can. 
private int voidcount;
private int VoidCount;
private int void_count;
private int void_Count;
private int voidCount;
Static Final Fields (Constants) Upper Snake Case Constants should be fully capitalized and before the start of a new word should be an underscore (_). It is very important that constants have descriptive but short names to describe the object/value that they will be holding. Any single letter constants should not be used even in the instance of loops. Constants should not start with $ or numbers even though they can. 
private static final int maxvoids = 5;
private static final int MaxVoids = 5;
private static final int max_voids = 5;
private static final int max_Voids = 5;
private static final int maxVoids = 5;
private static final int MAX_VOIDS = 5;

Modding-Specific Naming Conventions

Type Description
ModID The ModID must be all lowercase and unique. To make a ModID unique, it should be the name of the mod and optionally can contain the author's username. ModIDs can contain numbers, lowercase letters, underscores and dashes. An example of a valid ModID is largevoidmod or large_void_mod. An example of an invalid ModID is LargeVoidMod or 4g%*lgf'#~.
ResourceLocations and Registration Names ResourceLocations and Registration Names must be all lowercase and can only contain numbers, lowercase letters, underscores, dashes and forward slashes. Usually, new words are separated by an underscore. An example of a valid entry is void_apple or apples/void_apple. An example of an invalid entry is VoidApple or gf%*{}:.jdf.
Class Naming When constructing the name of a class, it is highly recommended to suffix the name with the type of object. For example if we wanted a void block, we could put VoidBlock. An unrecommended example of a class name would be BlockVoid.
